# 1.1 기술 부채란 무엇인가?

복잡한 소프트웨어 시스템을 만드는 모든 조직은 기술 부채를 가지고 있다.

### 📍 사례: 페이스북(메타)의 PHP 선택
* **초기:** 신속한 제품 출시를 위해 PHP 사용.
* **성장기:** 서비스 규모가 커지며 성능과 확장성 한계 직면.
* **해결:** 기존 코드를 버리는 대신 PHP 트랜스파일러를 만들어 기술 부채를 해결할 방법을 모색함.

---

## 💡 핵심 메시지
기술 부채를 절대 가져서는 '안 된다'는 것이 아니다. 핵심은 언제 어떻게 갚을지 모르는 상태에서 **부주의하게** 많은 부채를 가져서는 안 된다는 것이다.

### "최선은 선의 적이다"
* **의미:** 최선에 도달하려다 선에 도달하지 못한다.
* 모든 사항을 완벽히 이해하고 최상의 디자인이 나올 때까지 기다린다면 시장 점유율 확보 기회를 놓칠 뿐이다.
* **더 나은 방법:** 먼저 개발하고 사용자에게 선보여 피드백을 받는 것.

### 부채가 문제가 되는 지점
부채 자체는 문제가 아니다. 진짜 문제는 아래 세 가지가 없을 때 발생한다.
1. 부채를 제대로 **식별/인지**하지 않음.
2. 부채의 양을 **측정**하지 않음.
3. 부채를 **상환**하지 않음.

방치된 부채는 결국 불어나는 이자가 되어 시스템을 압도한다. 초기에는 머릿속에서 관리가 가능해 보여도, 결국 증가하는 복잡성을 감당할 수 없게 된다.
**코드, 테스트, 문서 등 기술적 산출물을 재구성하고 리팩터링하는 것은 보이지 않지만 매우 중요한 작업이다.**

## 📊 기술 부채 사분면 (그림 1.1)

![기술 부채 사분면](https://mblogthumb-phinf.pstatic.net/20150211_170/muchine98_1423619260300JtnY0_PNG/techDebtQuadrant.png?type=w2)

1. **무모함 + 의도적 ("We don't have time for design"):** 설계가 필요한 걸 알면서도 시간 핑계로 무시하고 코딩만 하는 경우. 아주 위험한 부채다.
2. **신중함 + 의도적 ("We must ship now and deal with consequences"):** 비즈니스 목표를 위해 전략적으로 부채를 지는 경우. 나중에 갚겠다는 명확한 계산이 서 있는 합리적 판단이다.
3. **무모함 + 무의식적 ("What's Layering?"):** 무엇이 문제인지조차 모르는 생존 모드. 숙련도 부족이나 잘못된 의사결정으로 인해 발생한다.
4. **신중함 + 무의식적 ("Now we know how we should have done it"):** 개발 후에야 더 나은 방법을 깨닫는 경우. 경험을 통해 배우는 과정에서 발생하는 자연스러운 부채다.

## 🔎 결론
미래에 부채를 분석하고 상환하기로 약속만 한다면, 전략적으로 부채를 떠맡는 것은 **완벽하게 합리적인 판단**이다.
